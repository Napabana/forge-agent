# 项目规划

### M1 阶段：核心事务机制与状态持久化

#### Task 1.1: SQLite 状态机与 DAG 引擎实现

* **任务描述：** 在项目根目录创建 `task/engine.py`。引入 `sqlite3` 标准库。编写 `TaskEngine` 类，在初始化时自动创建 `tasks`（字段包括：`task_id`, `status`, `owner`, `worktree_path`）和 `task_dependencies`（字段包括：`task_id`, `depends_on_id`）两张表。在连接初始化中执行 `PRAGMA journal_mode=WAL;` 开启预写式日志。编写方法实现：1. 插入新任务；2. 添加任务依赖关系；3. 查询某任务的所有前置依赖是否已全部处于完成状态。
* **产出：** `task/engine.py` 文件，包含完整的关系型表初始化及底层 SQL 增删改查封装逻辑。
* **目的：** 废弃原有的 JSONL 文本状态机，解决多智能体并发状态下的读写锁冲突，为后续的任务流转提供 ACID 事务保障。

#### Task 1.2: Git Worktree 事务上下文管理器

* **任务描述：** 创建 `runtime/worktree.py`。实现名为 `WorktreeSession` 的异步上下文管理器（实现 `__aenter__` 和 `__aexit__` 方法）。在 `__aenter__` 中，调用 `asyncio.create_subprocess_shell` 执行 `git worktree add <动态路径> <分支名>` 命令；在 `__aexit__` 中，无论是否抛出异常，必须强制执行 `git worktree remove --force <动态路径>` 并清理残余文件。
* **产出：** `runtime/worktree.py` 文件，包含强一致性的文件系统操作逻辑及完整的异常捕获机制。
* **目的：** 实现任务执行环境与主代码库的物理隔离，确保智能体修改代码崩溃或测试失败时，底层文件系统能够安全、干净地回滚。

#### Task 1.3: M1 阶段基线链路集成与 Mock 脚本

* **任务描述：** 编写 `scripts/m1_demo.py`。引入 `TaskEngine` 和 `WorktreeSession`。编写一段测试流程：初始化数据库 -> 插入一个测试任务 -> 使用 `async with WorktreeSession(...)` 开启隔离区 -> 在隔离区内模拟文件写入（模拟 Agent 行为）-> 抛出主动模拟的异常触发回滚 -> 验证隔离区被安全删除且数据库状态更新为失败。
* **产出：** `scripts/m1_demo.py` 测试脚本及执行日志。
* **目的：** 验证数据库事务与文件系统隔离机制的接口契约是否正确，为后续整合 LLM 决策主循环提供可靠的底层环境。

### M2 阶段：多智能体协作与 Harness 协议整合

#### Task 2.1: 单进程异步通信总线开发

* **任务描述：** 创建 `ipc/bus.py`。基于 Python 原生的 `asyncio.Queue` 实现 `AgentBus` 类。编写 `publish(topic: str, message: dict)` 方法用于消息推送，编写 `subscribe(topic: str)` 异步生成器方法用于消息监听与分发。设计标准的消息数据载体（Data Class），包含 `sender_id`, `receiver_id`, `action`, `payload` 等字段。
* **产出：** `ipc/bus.py` 文件，包含异步发布/订阅（Pub/Sub）机制的代码实现。
* **目的：** 替代 `s20` 框架中基于文件系统的轮询通信，实现单机环境下多智能体之间的低延迟消息同步。

#### Task 2.2: Hooks 与 Permission 拦截器逻辑移植

* **任务描述：** 从 s20 源码中提取工具执行的钩子与权限机制，在 `harness/` 目录下创建 `hooks.py` 和 `permission.py`。实现 `ToolExecutor` 包装器类，确保在任何 Shell 命令或文件读写工具执行前，必须先通过 `PermissionManager.check_access()` 校验（如拦截 `rm -rf` 等危险指令），并触发前置与后置事件日志记录。
* **产出：** `harness/hooks.py` 与 `harness/permission.py` 文件，包含工具链的安全拦截逻辑。
* **目的：** 规范智能体的执行边界，确保所有的底层工具调用都受到审计和授权控制，防止越权操作。

### M3 阶段：操作系统级沙箱与安全加固

#### Task 3.1: Docker API 资源隔离与系统安全封装

* **任务描述：** 重构现有的 `tools/runtime.py` (或 `forge-agent` 中的 `DockerRuntime` 类)。使用 `docker` Python SDK 修改容器启动参数。必须强制注入以下参数配置：`mem_limit='512m'`（或动态配置）、`nano_cpus=1000000000`（限制为单核）、`network_mode='none'`（禁用网络）、`read_only=True`（根文件系统只读），并将 `/tmp` 挂载为 `tmpfs`。
* **产出：** 更新后的 `tools/runtime.py` 运行时类代码。
* **目的：** 在宿主机内核层面严格限制 Agent 的计算资源占用，并阻断未经授权的外网访问，防止死循环代码或恶意脚本耗尽服务器资源。

#### Task 3.2: 基于 Worktree 的读写路径白名单配置

* **任务描述：** 修改 Task 3.1 中的 `DockerRuntime` 挂载（Volumes）逻辑。结合 Task 1.2 的 `WorktreeSession`，确保在挂载宿主机目录时，仅将当前任务动态生成的 Worktree 路径以读写 (`rw`) 模式挂载到容器内的特定工作目录，而将主代码库或其他依赖库设定为只读 (`ro`) 挂载。
* **产出：** `tools/runtime.py` 中的路径挂载与权限分配逻辑。
* **目的：** 实施最小权限原则，从物理隔离层面彻底杜绝 Agent 越权修改非目标任务文件的风险。

### M4 阶段：端到端验证与测试覆盖

#### Task 4.1: 核心机制的高覆盖率单元测试生成

* **任务描述：** 使用 `pytest` 与 `pytest-asyncio`，在 `tests/` 目录下生成针对前三个阶段的边界条件测试用例。必须包含：并发调用 SQLite 时的读写一致性测试、Worktree 强制中断（`KeyboardInterrupt` 模拟）后的目录清理断言测试、Docker 资源超载配置格式的校验测试。
* **产出：** `tests/` 目录下的一系列测试文件，要求核心逻辑（Engine, Worktree, Runtime, Bus）测试覆盖率达到 85% 以上。
* **目的：** 提供严谨的工程化交付证明，确保架构具备应对生产环境异常的健壮性。

#### Task 4.2: 整体 ReAct 主循环集成与日志回放

* **任务描述：** 在 `agent/core.py` 中，完成对底层 SQLite、异步总线、Docker 沙箱和 Git Worktree 的全量引入。编写完整的端到端测试脚本，通过输入一段自然语言的 Issue 描述，触发 ReAct 循环，记录并输出 JSONL 格式的完整事件日志（包含推理过程、工作树创建、命令执行和资源回收）。
* **产出：** `agent/core.py` 的最终集成代码与系统端到端执行的 JSONL 审计日志。
* **目的：** 实现全链路的闭环验证，提供可供面试演示或系统回放的工程级交付物。
